Implementação de um Sistema de Compra e Venda com Banco de Dados MSSQL

Sobre o Projeto

Este projeto, intitulado Implementação de um Sistema de Compra e Venda com Banco de Dados MSSQL, foi desenvolvido como parte da Missão Prática Nível 2 Mundo 3 do curso da Estácio de Desenvolvimento Full Stack. O objetivo principal é implementar um sistema de gerenciamento de compras e vendas de produtos utilizando um banco de dados MSSQL. O projeto aborda a modelagem de dados, criação de tabelas e relacionamentos, inserção de dados e a realização de consultas para extrair informações relevantes.

Características Principais

  • Modelagem de Banco de Dados Relacional: O projeto define entidades como Usuários, Pessoas (Físicas e Jurídicas), Produtos e Movimentos (compras e vendas), estabelecendo relacionamentos para garantir a integridade dos dados.
  • Criação e Gerenciamento do Banco de Dados: Utilização da linguagem SQL para criar o banco de dados com tabelas, restrições e relacionamentos.
  • Inserção de Dados: Adição de exemplos de dados em todas as tabelas para simular o funcionamento do sistema e permitir a realização de consultas.
  • Consultas SQL Complexas: Implementação de diversas consultas para extrair informações como dados de pessoas, movimentações de entrada e saída, valores totais, e relatórios por produto e operador.
  • Exploração de Conceitos Avançados: Utilização de sequências, chaves estrangeiras, junções entre tabelas e funções de agregação para consultas mais eficientes e informativas.

Tecnologias Utilizadas

  • MSSQL: Sistema gerenciador de banco de dados utilizado para armazenar e gerenciar os dados.
  • SQL Server Management Studio: Ferramenta para interagir com o banco, executar scripts e realizar consultas.
  • DBDesigner Fork: Software para a modelagem visual do banco de dados.

Aprendizados

O projeto permitiu explorar os conceitos de modelagem de banco de dados relacional, a sintaxe SQL para criação e manipulação de dados (DDL e DML), e vivenciar a experiência prática de implementar um sistema com banco de dados. O trabalho também aborda a importância de chaves estrangeiras, a diferença entre sequences e identity, e a utilização de operadores da álgebra e cálculo relacional para construir consultas eficientes.

    Tabelas Principais

    1. Usuários (Operadores)
      • ID_Usuario (chave primária): Identificador único para cada usuário (operador) do sistema.
      • Nome: Nome completo do usuário.
      • Email: Endereço de email do usuário. Deve ser único entre todos os usuários.
      • Senha: Senha para acesso ao sistema pelo usuário.
    1. Pessoas
      • ID_Pessoa (chave primária): Identificador único para cada pessoa, seja ela física ou jurídica.
      • Nome: Nome da pessoa (ou razão social, no caso de pessoa jurídica).
      • Telefone: Número de telefone de contato.
      • TipoPessoa: Indica o tipo da pessoa ('F' para física e 'J' para jurídica).
      • CPF_CNPJ: Número do CPF (para pessoa física) ou CNPJ (para pessoa jurídica). Deve ser único.
      • Logradouro: Endereço da pessoa, incluindo rua, número e complemento.
      • Cidade: Cidade do endereço.
      • Estado: Estado do endereço.
      • Email: Endereço de email da pessoa.
    1. Produtos
      • ID_Produto (chave primária): Identificador único de cada produto cadastrado no sistema.
      • Nome: Nome ou descrição do produto.
      • Quantidade: Quantidade disponível do produto em estoque.
      • PrecoVenda: Preço unitário de venda do produto.
    1. Movimento
      • ID_Movimento (chave primária): Identificador único de cada movimento de compra ou venda.
      • ID_Produto (chave estrangeira): Referência ao produto envolvido no movimento.
      • ID_Operador (chave estrangeira): Referência ao usuário que realizou o movimento.
      • ID_Pessoa (chave estrangeira): Referência à pessoa (física para vendas e jurídica para compras) envolvida no movimento.
      • TipoMovimento: Indica se o movimento é uma compra ('C') ou venda ('V').
      • Quantidade: Número de unidades do produto movimentadas.
      • PrecoUnitario: Preço unitário do produto no momento do movimento.
      • DataMovimento: Data e hora em que o movimento foi registrado.

    Relações

    • Usuários e Movimentos: A tabela Movimento associa cada operação (compra ou venda) a um usuário específico, permitindo rastrear quem realizou cada movimento.
    • Pessoas e Movimentos: A tabela Movimento também associa cada operação a uma pessoa específica, diferenciando se a operação foi uma compra (pessoa jurídica) ou venda (pessoa física).
    • Produtos e Movimentos: Cada movimento de compra ou venda está diretamente relacionado a um produto específico, impactando a quantidade disponível desse produto no estoque.

    Análise e Conclusão:

    “Como são implementadas as diferentes cardinalidades, basicamente
    1X1, 1XN ou NxN, em um banco de dados relacional?”

    • 1X1 (Um para Um): Ambas as tabelas possuem uma chave única que referencia a outra tabela. Exemplo: Uma pessoa possui apenas um passaporte e um passaporte pertence a apenas uma pessoa.
    • 1XN (Um para Muitos): Uma tabela possui uma chave única que referencia várias entradas em outra tabela. Exemplo: Um cliente pode fazer vários pedidos, mas um pedido pertence a apenas um cliente.
    • NXN (Muitos para Muitos): É necessário criar uma tabela intermediária para associar as duas tabelas principais, com chaves estrangeiras para ambas. Exemplo: Um aluno pode se matricular em vários cursos e um curso pode ter vários alunos.

    “Que tipo de relacionamento deve ser utilizado para representar o uso
    de herança em bancos de dados relacionais?”

    • Tabela única: Todas as subclasses compartilham uma única tabela com uma coluna para indicar o tipo.
    • Tabela por subclasse: Cada subclasse possui sua própria tabela com atributos específicos e uma chave estrangeira para a tabela pai.

    “Como o SQL Server Management Studio permite a melhoria da
    produtividade nas tarefas relacionadas ao gerenciamento do banco de
    dados?”

    • Interface gráfica: Simplifica tarefas como criação e gerenciamento de tabelas, consultas, backups e segurança.
    • Ferramentas de desenvolvimento: Editor de consultas com realce de sintaxe, depuração e execução de scripts.
    • Monitoramento e otimização: Análise de desempenho, identificação de gargalos e ajuste de consultas.
    • Automação de tarefas: Agendamento de jobs e scripts para automatizar rotinas de manutenção.

    Análise e Conclusão:

    “Quais as diferenças no uso de sequence e identity?”

    • Sequence: Gera números sequenciais, mas não está vinculada a uma tabela específica. Pode ser usada em várias tabelas ou até mesmo fora delas.
    • Identity: Vinculada a uma coluna específica de uma tabela. Gera automaticamente valores únicos para essa coluna quando novos registros são inseridos.

    “Qual a importância das chaves estrangerias para a consistência do banco?”

    Garante a consistência dos dados entre tabelas, evitando a criação de registros órfãos (sem correspondência na tabela relacionada). Basicamente, mantém tudo organizado e conectado.

    “Quais operadores do SQL pertencem à álgebra relacional e quais são definidos no cálculo relacional?”

    • Álgebra Relacional: SELECT (com restrições), PROJECT, UNION, INTERSECT, EXCEPT, JOIN, RENAME.
    • Cálculo Relacional: Operadores lógicos como AND, OR, NOT.

    “Como é feito o agrupamento em consultas, e qual requisito é
    obrigatório?”

    Usado para agrupar linhas com valores iguais em certas colunas. O comando GROUP BY é usado para isso.

    Requisito obrigatório: A cláusula SELECT deve incluir apenas colunas que estão sendo agrupadas ou funções de agregação (como SUM, AVG, etc.).
